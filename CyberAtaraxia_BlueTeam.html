Phase 0 · Module 01
Security Fundamentals
CIA triad · Security controls · OSI model · Blue Team roles · Physical security
The CIA Triad — the foundation of everything

Every security control, detection rule, and incident response action exists to protect one of three properties: Confidentiality (only authorized parties access data), Integrity (data is not modified without authorization), and Availability (systems are accessible when needed). When you investigate an incident, always ask: which property was violated?

Confidentiality
Encryption, access control, MFA, DLP. Violated by: data exfiltration, credential theft, overpermissioning.
Integrity
Hashing, digital signatures, file integrity monitoring. Violated by: ransomware, log tampering, SQL injection, timestomping.
Availability
Redundancy, backups, DDoS protection, patching. Violated by: ransomware encryption, DDoS, destructive wiper malware.

TCP 3-way handshake and what deviations mean

Every TCP connection: SYN → SYN-ACK → ACK. Forensically: SYN with no SYN-ACK = port closed/filtered. Sequential SYN to many ports from one IP = port scan (Nmap). Regular SYN to one IP:port at fixed intervals = C2 beaconing. Flood of SYN to one port = DoS. RST (reset) mid-connection = abrupt termination, common in scan activity or blocked connections.

Network devices as detection sources
Firewall
Logs allow/block decisions by IP, port, protocol. Source of NetFlow. Detects: port scans, C2 beaconing, unexpected outbound connections.
IDS/IPS (Snort/Suricata)
Inspects packet payload against signatures. IDS = alert only. IPS = alert + block. Detects: exploit attempts, malware callbacks, protocol anomalies.
Proxy server
Logs all HTTP/HTTPS requests: URL, user-agent, referrer, response code. Invaluable for tracing malware downloads, C2, and exfiltration.
DNS server
Logs every query and response. Essential for detecting C2 via DNS, DGA, tunneling, and compromised hosts making suspicious lookups.

Phase 0 · Module 03
Active Directory
AD structure · Kerberos · AD attack techniques · Detection
Why AD is the crown jewel

Active Directory is Microsoft's directory service managing users, computers, groups, and policies in virtually every enterprise environment. It is the central authentication and authorization system — compromising AD equals compromising the entire organization. Domain Admin access lets an attacker create accounts, exfiltrate everything, deploy ransomware to all machines, and maintain indefinite access.

Domain Controller (DC)
Runs AD. Stores all accounts and password hashes (NTDS.dit). The ultimate attacker target. Always monitor DC logs with highest priority.
Service Principal Names (SPNs)
Identifiers for services in AD. Any domain user can request service tickets for any SPN — exploited in Kerberoasting to get crackable hashes offline.
Privileged groups
Domain Admins, Enterprise Admins, Schema Admins. Any EID 4732 adding to these groups is critical — always alert on these events unconditionally.

Phase 1 · Module 04
SOC Architecture & Operations
Tier roles · SIEM · SOAR · EDR · Ticketing · Shift handover
SOC tier structure
Tier 1 — Analyst
Monitors queue 24/7. Initial triage: FP vs TP. Enriches with OSINT. Escalates to Tier 2. High volume, lower complexity investigations.
Tier 2 — IR Specialist
Receives Tier 1 escalations. Deep investigation, scoping, containment decisions. Broader tool access, forensic skills. Interfaces with stakeholders during incidents.
Tier 3 — Threat Hunter
Proactively hunts threats that no alert catches. Develops new detection rules, deep malware analysis, advises on ATT&CK coverage gaps and blind spots.
SOC toolchain
SIEM
Collects, normalizes, correlates logs. Fires alerts based on detection rules. Splunk, Microsoft Sentinel, IBM QRadar, Wazuh (open source).
SOAR
Automates response actions (isolate host, block IP, create ticket). Splunk SOAR, Palo Alto XSOAR. SIEM detects; SOAR responds.
EDR
Endpoint telemetry: process tree, file writes, network connections, memory activity. CrowdStrike, SentinelOne, Carbon Black, Defender for Endpoint.
Case Management
Tracks incidents from detection to closure. TheHive5 (open source, BTL1 exam-relevant), Jira, ServiceNow. Every investigated alert needs a case.

Phase 1 · Module 05
MITRE ATT&CK Framework
14 tactics · Techniques · Pyramid of Pain · Cyber Kill Chain · APT attribution
Three complementary frameworks
Cyber Kill Chain (Lockheed)
7 linear phases: Reconnaissance → Weaponization → Delivery → Exploitation → Installation → C2 → Actions on Objectives. Useful for understanding attack progression.
MITRE ATT&CK
14 tactics, hundreds of techniques based on real-world threat actor observations. The standard language for describing attacks. Maps TTPs to detection opportunities.
Pyramid of Pain (Bianco)
Shows detection difficulty vs pain to attacker. Bottom = hashes/IPs (easy to change). Top = TTPs (forces complete retool). Detect at TTP level for maximum impact.
The 14 ATT&CK tactics (memorize these)
TA0043 Reconnaissance
Gathering info before the attack: OSINT, scanning, phishing for credentials/info.
TA0001 Initial Access
Getting in: phishing, exploit public apps, valid accounts, supply chain compromise.
TA0002 Execution
Running malicious code: PowerShell, WMI, scheduled tasks, mshta, scripts.
TA0003 Persistence
Surviving reboots: Run keys, scheduled tasks, services, startup folder, DLL hijacking.
TA0004 Privilege Escalation
Gaining higher privileges: UAC bypass, token manipulation, local exploits.
TA0005 Defense Evasion
Avoiding detection: obfuscation, disabling AV, timestomping, LOLBins, clearing logs.
TA0006 Credential Access
Stealing creds: LSASS dumping, Kerberoasting, keylogging, SAM/NTDS.dit access.
TA0007 Discovery
Learning the environment: net commands, BloodHound, ping sweeps, SPN enumeration.
TA0008 Lateral Movement
Moving between hosts: PtH, PsExec, RDP, WMI remote execution, SSH.
TA0009 Collection
Gathering data to exfiltrate: screen capture, clipboard, file staging, email collection.
TA0011 C2
Communicating back: DNS tunneling, HTTPS C2, beaconing, Cobalt Strike, web shells.
TA0010 Exfiltration
Getting data out: C2 channel, cloud storage, encrypted archive, DNS tunneling.
TA0040 Impact
Destroying/disrupting: ransomware, wiper malware, defacement, DoS, data destruction.
TA0042 Resource Dev
Building attack infra: acquiring domains, setting up C2 servers, developing malware.

Phase 1 · Module 06
Threat Intelligence
IOCs · TI lifecycle · OSINT platforms · MISP · ISACs · PAP/TLP
Intelligence types
Tactical (IOCs)
File hashes, IPs, domains, URLs. Immediately actionable — block or hunt now. Short shelf life: IPs rotate in hours, domains in days.
Operational (TTPs)
How a specific actor operates: tools, techniques, target selection. Longer-lived than IOCs. Maps directly to ATT&CK techniques.
Strategic
High-level for executives: which threat actors target your industry, geopolitical threat drivers, industry risk trends.
PAP and TLP

TLP (Traffic Light Protocol) governs information SHARING — who you can share intelligence with. PAP (Permissible Actions Protocol) governs what ACTIONS you can take with the intelligence you receive. Both appear on shared threat reports from ISACs and vendors.

TLP:WHITE / TLP:CLEAR
No restrictions. Can be shared publicly.
TLP:GREEN
Share within your community/sector. Not for public release.
TLP:AMBER
Share only within your organization and directly with need-to-know partners.
TLP:RED
No sharing. For recipient eyes only. Cannot be used in automated systems.
Key OSINT platforms for analysts
VirusTotal
Submit hash, IP, domain, or file. 70+ AV engines. Use the Relationships graph to pivot to connected infrastructure and campaigns.
AlienVault OTX
Community TI. Search any IOC to find "pulses" — threat reports linking it to campaigns and threat actors. Subscribe to sector feeds.
abuse.ch
URLhaus (malicious URLs), MalwareBazaar (malware samples), ThreatFox (IOCs), Feodo Tracker (C2 infrastructure). Free API access.
MISP
Open source TI platform for sharing and correlating IOCs within and between organizations. Used by ISACs and national CERTs globally.

Linux forensic log locations
/var/log/auth.log
SSH logins, sudo, su. Primary auth source on Debian/Ubuntu. Key strings: "Invalid user", "Failed password", "Accepted publickey from".

Same as auth.log on RHEL/CentOS. SSH brute force, failed sudo attempts, PAM module activity.

Login history (last command) and failed logins (lastb). Session timeline including source IPs and durations.

Phase 1 · Module 08
Alert Management & Triage
Triage workflow · FP vs TP · Multi-alert correlation · Documentation
6-step triage workflow
1
Receive alert
Alert fires in SIEM, ticket auto-opens in TheHive or Jira. Never dismiss without investigation — even if it looks like noise.
2
Gather context
Who is the user? What asset? What time? How often has this rule fired before on this asset in the past 30 days?
3
Enrich
OSINT the IPs/domains/hashes on VirusTotal and OTX. Check asset inventory and user's recent activity history in the SIEM.
4
Correlate
Are there other alerts on the same host/user in the same time window? One alert = possible noise. Three correlated alerts = pattern.
5
Decide
FP: close with written explanation of WHY. TP: escalate to Tier 2 with all enrichment and correlation data attached.
6
Document
Every decision needs a written record. Your notes protect you legally, enable shift handover, and build institutional knowledge.

Splunk SPL essentials

Sigma rules — platform-agnostic detection

Sigma is a generic signature format: write once, convert to SPL, KQL, Elasticsearch, or any SIEM. The exam tests your ability to read and interpret Sigma rules. Key fields: detection.selection defines what to match, condition applies logic, logsource specifies the data source.

# Sigma rule — Mimikatz detection via process name / arguments title: Mimikatz Credential Dumping status: stable logsource: category: process_creation product: windows detection: selection: Image|endswith: '\mimikatz.exe' CommandLine|contains: - 'sekurlsa' - 'lsadump' - 'privilege::debug' condition: selection level: critical tags: - attack.credential_access - attack.t1003.001
Alert tuning principles
low risk
Exclude specific known-good IPs
Precise, documented, easy to review. Revisit if the IP changes ownership or role.
medium risk
Exclude entire subnets
Only acceptable with a compensating detection covering that subnet. Document the compensating control.
never do this
Exclude privileged accounts to reduce noise
Attackers target exactly these accounts. Suppressing Domain Admins creates a fatal blind spot in the most critical targets.

Phase 2 · Module 10
Network Security Monitoring
Wireshark · PCAP analysis · C2 detection · DNS tunneling · Lateral movement
Wireshark display filters for analysts
// Protocol filters — start broad, then narrow http dns smb ftp ssh tls // Filter by IP address ip.addr == 192.168.1.100 ip.src == 10.0.0.5 && ip.dst == 185.220.101.45 // Detect large POST requests — possible data exfiltration http.request.method == "POST" && http.content_length > 1000 // DNS tunneling indicator — unusually long query names (>50 chars) dns.qry.name.len > 50 // Find all SYN packets to a specific destination (beaconing detection) tcp.flags.syn == 1 && ip.dst == 185.220.101.45 // SMB lateral movement — connections to admin shares smb.path contains "ADMIN$" OR smb.path contains "C$"
Suspicious traffic patterns to recognize
C2 Beaconing
Regular-interval connections to the same destination. Machine-automated = perfectly regular timing (e.g. exactly every 60s). Human browsing is irregular. Timing regularity is the key detection signal.
DNS Tunneling
Data encoded as base64 subdomain labels: "aGVsbG8gd29ybGQ.evil.com". Indicators: query names >50 chars, high frequency, base64 charset in subdomains, replies with TXT records containing encoded data.
Data Exfiltration
Large outbound transfers to unusual destinations. HTTPS to IP addresses (not domain names) is suspicious — most legitimate cloud services use hostnames. Look for unusual volume at unusual times.
SMB Lateral Movement
Workstation-to-workstation SMB connections to ADMIN$ or C$. Normal workstations should not connect directly to each other via SMB — only to file servers.

Phase 2 · Module 11
IDS: Snort & Suricata
Rule writing · Signature detection · Anomaly detection · Alert analysis
IDS vs IPS

An IDS (Intrusion Detection System) passively monitors and alerts — it does not block. An IPS (Intrusion Prevention System) sits inline and can drop malicious packets in real time. Both Snort and Suricata can run in either mode. In a SOC, IDS mode feeds your SIEM with telemetry; IPS mode provides active blocking. The trade-off: IPS requires higher confidence rules to avoid blocking legitimate traffic.

Signature-based detection
Matches traffic against known attack patterns. Fast, precise for known threats. Blind to novel zero-days. Requires regular rule updates (Snort VRT, Emerging Threats Open).
Anomaly-based detection
Establishes a traffic baseline, flags statistical deviations. Can detect novel attacks but generates more false positives. Example: unusual protocol on a standard port.
Snort rule anatomy
# Structure: action proto src_ip src_port -> dst_ip dst_port (options) # Detect Mimikatz LSASS access over network alert tcp any any -> $HOME_NET 445 (msg:"ET CREDS Mimikatz LSASS Access"; flow:established,to_server; content:"|60 48 8b 05|"; depth:4; classtype:credential-theft; sid:2024001; rev:1;) # Detect DNS tunneling — long subdomain query alert dns any any -> any 53 (msg:"Possible DNS Tunneling - Long Query"; dns.query; pcre:"/[A-Za-z0-9+\/]{40,}\./i"; sid:2024002; rev:1;)

Key fields: sid = unique rule ID, msg = human-readable alert name for your SIEM, content = byte pattern to match in payload, pcre = regex pattern, classtype = alert category for grouping.

Phase 3 · Module 12
Phishing Analysis
Email headers · SPF/DKIM/DMARC · VBA macro analysis · Sandbox · BEC
Email header analysis

The Received chain shows every mail server the message passed through in reverse order (bottom = origin, top = last hop). The From header is trivially spoofed. Always check: Return-Path (true bounce address), X-Originating-IP (origin IP), and Authentication-Results (SPF/DKIM/DMARC outcomes).

RecordFunctionFailure means
SPFLists authorized sending IPs for a domain. Receiver checks if sending IP is listed.Email from unauthorized server — likely spoofed or misdirected
DKIMCryptographic signature on message content. Receiver verifies via public key in DNS.Email was modified in transit or not signed by the claimed domain
DMARCPolicy defining what to do when SPF/DKIM fail: reject, quarantine, or none (monitor).Domain owner's policy tells receivers to reject/quarantine the message
VBA macro analysis — extracting malicious code

Phishing emails frequently deliver macro-enabled Office documents (.doc, .xlsm). The macro executes on open and runs a PowerShell payload. Analyzing VBA without executing the file requires extracting and reading the macro code statically.

Phase 3 · Module 13
Malware Analysis
Static PE analysis · Import table · High entropy · Dynamic analysis · IOC extraction
PE file structure and what to look for

A PE (Portable Executable) file has a header, sections (.text for code, .data for variables, .rsrc for resources), and an Import Address Table (IAT) listing every Windows API function the malware calls. The IAT reveals capabilities before execution — this is the most valuable part of static analysis.

Suspicious importCapability indicated
CreateRemoteThread + WriteProcessMemoryProcess injection — injecting shellcode into another running process
VirtualAllocEx + OpenProcessMemory allocation in remote process — classic shellcode injection pattern
InternetConnect / URLDownloadToFileC2 communication or payload download
RegSetValueEx / RegCreateKeyExRegistry persistence — writing Run keys or other persistence mechanisms
CryptEncrypt / BCryptEncryptFile encryption capability — ransomware or protected C2 communication
SetWindowsHookExKeyboard/mouse hooking — keylogger capability
Section entropy — detecting packing

Normal code sections have entropy around 5.0–6.5. Entropy above 7.0 in the .text section indicates the code is packed, encrypted, or obfuscated. The actual payload is concealed and decrypts itself at runtime. You must use a sandbox or debugger to unpack and reveal the true functionality. Tool: DIE (Detect It Easy) or PEStudio both show section entropy.

Static analysis workflow
1
Hash the file
MD5 + SHA256. Check VirusTotal immediately. Note detection rate, first submission date, and associated campaigns.
2
Detect packing/obfuscation
Run DIE (Detect It Easy). High .text entropy (>7.0) = packed. Few meaningful strings = obfuscated. Both indicate dynamic analysis will be needed.
3
Extract strings
Use strings or FLOSS (for obfuscated strings). Hunt for: hardcoded IPs, domains, URLs, registry paths, mutex names, file paths, PDB debug paths.
4
Analyze PE imports (IAT)
Open in PEStudio or CFF Explorer. The import table reveals capabilities without running anything. See table above.
5
Document IOCs
File hash, hardcoded IPs/domains, mutex names, registry keys, file paths, PDB paths. Feed into SIEM and threat intel platform.

1

Phase 4 · Module 17
Memory Analysis With Volatility
Volatility 3 · Fileless malware · Process hiding · Code injection · Network state
Why memory forensics is critical

Memory forensics is the only way to detect fileless malware — attacks that live entirely in RAM and never write a file to disk. PowerShell Empire, Cobalt Strike Beacon, and many modern RATs operate this way. The only evidence is in a memory dump. Memory analysis also reveals: running processes (including kernel-hidden ones), network connections at the exact moment of capture, injected shellcode, decryption keys in plaintext, and cached credentials.

Volatility 3 essential commands
# List processes from the Windows process list (PsActiveProcessList) vol.py -f memory.raw windows.pslist # Scan physical memory for EPROCESS structures (finds hidden processes) vol.py -f memory.raw windows.psscan # KEY: Processes in psscan but NOT pslist = rootkit hiding them # Active network connections at capture time vol.py -f memory.raw windows.netstat # Find injected code in process memory (critical for fileless detection) vol.py -f memory.raw windows.malfind # List DLLs loaded by a specific process vol.py -f memory.raw windows.dlllist --pid 1234 # Extract command line arguments for all processes vol.py -f memory.raw windows.cmdline # Dump a suspicious process to disk for static analysis vol.py -f memory.raw windows.dumpfiles --pid 1234

Incident Response
PICERL lifecycle · Severity · Evidence collection · Containment · TheHive
PICERL — the 6-phase IR lifecycle
1
Preparation
Playbooks, tools, communication channels, escalation paths, retainer agreements — all defined BEFORE an incident. IR without preparation is chaos. TheHive5 configured, runbooks published.
2
Identification (Detection & Analysis)
SIEM alert, EDR detection, or user report triggers IR. Classify severity (P1/P2/P3), determine scope, identify affected assets and data.
3
Containment
Short-term: isolate infected systems from the network. Long-term: patch the exploited vulnerability. NEVER eradicate before fully scoping — you'll alert the attacker and lose visibility.
4
Eradication
Remove all malware, delete attacker accounts, close every persistence mechanism. Verify completely — missed persistence = re-infection within days.
5
Recovery
Restore from verified clean backups. Verify integrity before reconnecting. Monitor closely for 30 days post-recovery — recurrence is common if eradication was incomplete.
6
Lessons Learned
Post-incident review within 2 weeks while events are fresh. What detection failed? What response was slow? Update playbooks, create new SIEM rules from TTPs observed.
Evidence collection order (order of volatility)
1st — Volatile (collect first)
RAM dump · Active network connections · Running processes · Open file handles · Logged-in users. Lost on shutdown.
2nd — Semi-volatile
Event logs (before clearing) · Temp files · Browser cache · Registry hives in memory.
3rd — Non-volatile (disk)
Bit-for-bit disk image with write blocker · Stored logs · Registry files · File system artifacts.
Chain of custody: every piece of evidence must be documented — what it is, when collected, by whom, hash value, and every person who touched it. Without this, evidence may be inadmissible in legal proceedings.

SOC & Alert Management
What is the difference between SIEM and SOAR?
SIEM (Security Information & Event Management) collects, normalizes, and correlates logs to detect threats and fire alerts. SOAR (Security Orchestration, Automation & Response) automates the response to those alerts — e.g. auto-isolate a host when ransomware is detected. SIEM detects; SOAR responds. They complement each other: SIEM is the brain, SOAR is the hands.
Name the 3 SOC tier roles and their core responsibilities.
Tier 1 (Analyst): 24/7 queue monitoring, initial triage (FP vs TP), OSINT enrichment, escalation to Tier 2. Tier 2 (IR Specialist): deep investigation, scoping, containment decisions, forensic analysis, stakeholder communication. Tier 3 (Threat Hunter): proactive hunting for undetected threats, new detection rule development, deep malware analysis, coverage gap analysis using ATT&CK.
What must a shift handover document always include?
All open tickets and current status, active incidents or ongoing escalations, alerts paused pending enrichment, known FPs that need tuning, any environmental changes affecting monitoring (maintenance windows, new systems onboarded), and any threat intelligence received during the shift.
MITRE ATT&CK & Threat Intelligence
Name the 14 ATT&CK tactics in order.
Reconnaissance → Resource Development → Initial Access → Execution → Persistence → Privilege Escalation → Defense Evasion → Credential Access → Discovery → Lateral Movement → Collection → Command & Control → Exfiltration → Impact. Memory trick: "Recon, Resources, In, Ex, Persist, Priv, Def, Cred, Disc, Lat, Collect, C2, Exfil, Impact".
What is the Pyramid of Pain and why does it drive detection strategy?
The Pyramid of Pain (David Bianco) ranks IOC types by how much pain detecting them causes the attacker. Bottom (easy to change): hashes, IPs, domains. Middle: artifacts, tools. Top (hard to change): TTPs. Detecting and responding to TTPs — the ATT&CK technique layer — forces the attacker to completely retool their entire operation. This is why ATT&CK-based detection is more durable than IOC-based blocking.
What is TLP vs PAP?
TLP (Traffic Light Protocol) governs information SHARING — who you can share threat intel with: WHITE (public), GREEN (community), AMBER (organization + limited partners), RED (recipient only). PAP (Permissible Actions Protocol) governs what ACTIONS you can take with received intel: WHITE (unrestricted), GREEN (community sharing + detection), AMBER (internal use + detection), RED (cannot be used in automated systems). Both appear on threat reports from ISACs and vendors.
Log Analysis & SIEM
What is Windows EID 1102 and why does it always require escalation?
EID 1102 means the Windows Security event log was cleared. This is an attacker anti-forensics indicator — they are actively destroying evidence of their activity. A legitimate admin clearing logs would have a documented change ticket. Any EID 1102 without a prior-approved change ticket requires immediate escalation, and you should correlate all activity on that system in the preceding hours before the clear.
Password spraying vs brute force — how do you distinguish them?
Brute force: many passwords tried against ONE account — triggers lockout quickly. Password spraying: ONE or few passwords tried against MANY accounts — stays under per-account lockout thresholds. Detection signature: brute force = many EID 4625 for the same username from one IP. Spraying = many EID 4625 for many different usernames from one source IP, with only 1-2 attempts per account.
What is a Sigma rule and why is it useful for detection engineering?
Sigma is a generic, platform-agnostic detection rule format. Write once, convert to SPL (Splunk), KQL (Sentinel), Elasticsearch Query, or any other SIEM query language using the sigmac converter. This enables detection rules to be shared across the security community and applied regardless of which SIEM the organization uses. SigmaHQ maintains thousands of community rules mapped to ATT&CK techniques. Key fields: logsource (where to look), detection.selection (what to match), condition (logic).
Digital Forensics & IR
What is the correct order of evidence collection?
Most volatile first: (1) CPU registers/cache (rarely practical), (2) RAM memory dump, (3) Active network connections, (4) Running processes and open files, (5) Event logs before clearing, (6) Disk image (bit-for-bit with write blocker). The rule: collect what will be lost first. RAM is gone on shutdown; disk survives power cycling.
What is MACB and how do you detect timestomping?
MACB = Modified (last content change), Accessed (last read/executed), Changed ($MFT metadata changed), Born (file created on this volume). Timestomping = attacker modifies $STANDARD_INFORMATION timestamps to hide a file or make it appear legitimate. Detection: compare $STANDARD_INFORMATION vs $FILE_NAME timestamps in the MFT. $FILE_NAME is set by the kernel and cannot be modified without special tools. A mismatch = timestomping. Born timestamp newer than Modified = classic indicator.
Name the 6 PICERL phases in order.
Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned. Critical exam point: Containment happens BEFORE Eradication. You must stop the bleeding and fully understand the scope before cleaning up — premature eradication alerts the attacker and destroys your ability to understand the full compromise.
Network, Malware & Active Directory
What are LOLBins? Give 5 examples.
LOLBins (Living Off the Land Binaries) are legitimate Windows binaries abused by attackers for malicious purposes. They are signed by Microsoft, which may allow them to bypass AV and application whitelisting. Five examples: (1) certutil.exe — download files (-urlcache -split -f), decode base64, (2) mshta.exe — execute remote HTA scripts from a URL, (3) rundll32.exe — execute DLL payloads, (4) regsvr32.exe — Squiblydoo technique to execute remote scripts and bypass AppLocker, (5) bitsadmin — download files and create persistent BITS jobs.
What is Kerberoasting and how do you detect it?
Kerberoasting (T1558.003): any authenticated domain user can request Kerberos TGS service tickets for any account with an SPN. The ticket is encrypted with the service account's password hash. The attacker collects these tickets and cracks them offline at unlimited speed without touching the domain controller again. Detection: mass EID 4769 (Kerberos TGS request) events from a single workstation targeting many different service accounts (SPNs) in a short time window — especially for RC4 encryption type (0x17) which is weaker and easier to crack.
What does high PE section entropy indicate in malware analysis?
Normal code sections have entropy around 5.0–6.5. Entropy above 7.0 in the .text or another executable section indicates the section contains packed, encrypted, or obfuscated data. The actual malicious code is concealed and decrypts itself at runtime. This is used by attackers to evade static AV signatures. When you see high entropy, static analysis will be limited — you must use a sandbox (Any.run) or debugger to unpack and analyze the actual payload. Tool: DIE (Detect It Easy) shows per-section entropy values.
